## HIGH

### P2H1. POS tables (tickets, ticket_items, payments, shifts, shift_movements) have permissive RLS ✅
`supabase/migrations/20260428200000_create_tickets.sql:137-174`

Every authenticated session can read, insert, update, and delete *any* row in those tables. The four policies generated by the DO block are uniformly:

```sql
FOR SELECT TO authenticated USING (auth.uid() IS NOT NULL)
FOR INSERT TO authenticated WITH CHECK (auth.uid() IS NOT NULL)
FOR UPDATE TO authenticated USING (auth.uid() IS NOT NULL) WITH CHECK (auth.uid() IS NOT NULL)
FOR DELETE TO authenticated USING (auth.uid() IS NOT NULL)
```

The migration self-documents this as deliberate ("RLS is permissive (any authenticated user) at this phase; manager-gated paths land with the staff-roles wiring in a later migration" — lines 10-12). So it's a known limitation, not a hidden defect. But it's also the largest open RLS hole in the schema and worth re-flagging since you now have a real prod database with revenue records inside it. Right now any user who can authenticate (and that's the same pool that has the anon key in their browser bundle) can `DELETE FROM payments` or `UPDATE tickets SET paid_cents = 0`.

This is the same pattern as prior audit's **H5** but covering financial tables instead of `system_state` / `push_subscriptions`. Combined with **C2** (the hardcoded admin PIN being the only thing gating "admin mode"), the practical access boundary today is "anyone with a Supabase login" → "everything."

**Fix:** the migration's own comment names the design — staff-roles wiring. Concretely: add a `role` column (or a `staff_role` table mapping `auth.uid()` → role), rewrite the four policies per table to consult that role, and require manager-role for DELETE/UPDATE on payments and shift_movements. The SELECT policies can probably stay open across cashiers; INSERT and UPDATE of `tickets` / `ticket_items` should stay open during a shift (every cashier needs them). The destructive operations are what need narrowing.

### P2H2. Tab-visibility reconciliation does not use `withRetry` ✅
`src/state/AppContext.tsx:420` (and surrounding `reconcileAppts`)

When the tab regains focus, the app re-fetches all appointments to catch realtime events missed while backgrounded. Unlike every other DB read in the file (which goes through `withRetry`), this one fetch is a bare call:

```ts
const { data, error } = await supabase.from('appointments').select('*');
if (error || !data) return;
```

A transient blip on focus-return (very common: laptop wakes from sleep, the WebSocket reconnects but the first HTTP fetch occasionally fails) silently fails and the tab stays stale until the next focus event. The user has no idea anything went wrong — there's no syncError surfaced.

**Fix:** wrap the fetch:
```ts
const { data, error } = await withRetry(() => supabase.from('appointments').select('*'));
```
…and surface `syncError` if `error` is still set after the retries. Also consider also reconciling `queue_entries` and `completed_services` on visibility-return — the missed-events problem is identical for those tables; appointments was just the one that bit hardest.

## MEDIUM

### P2M1. Realtime payload shapes are cast as `Record<string, unknown>` without validation ◻
`src/state/AppContext.tsx` — 11 `postgres_changes` handlers in the realtime setup block (~lines 960-1110)

Every handler does roughly:

```ts
.on('postgres_changes', { event: '*', schema: 'public', table: 'manicurists' }, (payload) => {
  dispatch({ type: 'REMOTE_MANICURIST_UPSERT',
    manicurist: mapDbManicurist(payload.new as Record<string, unknown>) });
})
```

The `as Record<string, unknown>` cast skips shape validation. If a column is dropped or renamed in a later migration before the client is redeployed, `mapDbManicurist` will read `undefined`, fall back through its `??` chain, and produce a partially-empty row that flows straight into the reducer. The UI will then show a manicurist with no name/no color/etc., and the user will report "weird state" without an obvious cause.

This is latent today because the schema is stable, but it's a tripwire for the next schema change. Two safer patterns: a runtime shape check in each map function that returns `null` (and the handler skips the dispatch if so), or — heavier but cleaner — a single `validatePayload<T>(row, keys)` helper used by all 11 handlers.

### P2M2. `withRetry` retries every error, including permanent 4xx errors ✅
`src/state/AppContext.tsx:1217` (the `withRetry` definition)

```ts
async function withRetry<T>(fn, retries = 3, delayMs = 2000) {
  for (let attempt = 1; attempt <= retries; attempt++) {
    const result = await fn();
    if (!result.error) return result;
    if (attempt < retries) await new Promise(res => setTimeout(res, delayMs * attempt));
  }
  return fn();
}
```

A `401 invalid token`, `403 RLS denied`, `404 not found`, or `422 validation` is just as much "error" to this loop as a transient `503` — so the wrapper retries them three times, with backoff. From the user's perspective that means "Saving…" sits there for ~12 seconds before the failure toast finally fires. From the server's perspective it's 3× the load for unrecoverable requests. And on 401s specifically, the next retry's auth header is the same expired token, so they all fail identically.

**Fix:** branch on the error shape — Supabase errors have `code` and `status` (PostgREST). Retry only when the error is transient (network, no `code`, or status 5xx / 408 / 425 / 429). For everything else, return immediately. Keep the backoff for the transient branch.

### P2M3. `Date.now()` and `crypto.randomUUID()` inside the reducer ◻
`src/state/reducer.ts` — multiple cases. `Date.now()` at lines 213, 242, 381, 490, 549, 793, 1006. `crypto.randomUUID()` at line 136 inside `synthWalkInAppt`, which is called from `ASSIGN_CLIENT`, `REQUEST_ASSIGN`, and `SPLIT_AND_ASSIGN`.

A reducer is supposed to be a pure function of `(state, action) → state`. Both of these break determinism: replaying the same dispatch sequence produces different state. The practical consequences are limited today (no replay-based debug tooling, no event-sourcing, no SSR) but it does mean:

1. Tests can't simply call the reducer with a sequence of actions and assert on the result — they'd need to mock `Date.now` and `crypto.randomUUID` first.
2. The "did this dispatch actually change state" check the realtime echo prevention partly relies on becomes hairier — two identical UPSERTs produce different `lastEditedAt` values, so the reducer always reports "yes, state changed" even when functionally nothing did.
3. Time-travel debugging in React DevTools doesn't quite work as documented.

**Fix:** mint timestamps and UUIDs at the dispatch site (the component or callback that fires the action), pass them through the action. The reducer just consumes them. E.g.:

```ts
// before
dispatch({ type: 'CLOCK_IN', id: m.id });
// reducer: { ...m, clockInTime: Date.now() }

// after
dispatch({ type: 'CLOCK_IN', id: m.id, now: Date.now() });
// reducer: { ...m, clockInTime: action.now }
```

This is a "do it once for hygiene" refactor; not urgent.

### P2M4. `nightly-save-history` upsert mints a new `id` on every run ✅
`supabase/functions/nightly-save-history/index.ts:204-209`

```ts
const { error: upsertError } = await supabase
  .from("daily_history")
  .upsert({ id: crypto.randomUUID(), date: todayLA, entries },
          { onConflict: "date" });
```

`daily_history.id` is the primary key (`text PRIMARY KEY` per `20260405100000_create_daily_history.sql:19`). Supabase's `.upsert({...}, { onConflict: 'date' })` updates *every column* of the matched row, including `id`. So the second time the function runs for the same `date`, the PK value of that row changes.

Impact today is small: nothing FKs into `daily_history`, and the client reads it by `date` not `id`. But it's pointlessly destructive of any external reference (e.g., a future sync that snapshots IDs, or a logger that records "today's history is row X"). It's also a strict-mode-no-no that PostgREST won't warn you about because the operation succeeds.

**Fix:** drop the `id` field from the upsert payload entirely. The column has no DEFAULT, so you'll also need to either give it `DEFAULT gen_random_uuid()::text` in a tiny migration, or compute the id only on initial INSERT and use a SQL function for that. Easiest: change the schema to `id uuid PRIMARY KEY DEFAULT gen_random_uuid()` (a `text → uuid` cast, the existing string IDs cast fine), then `.upsert({ date, entries }, { onConflict: 'date' })` and let Postgres handle PK assignment.

## LOW

### P2L1. `reconcileAppts` silently overwrites local appointments with a fresh DB snapshot ◻
`src/state/AppContext.tsx:420-440` (`reconcileAppts`)

On visibility-return it reads `from('appointments').select('*')` and then `dispatch({ type: 'LOAD_STATE', state: { appointments: fresh } })` — full replace, no merge, no `updated_at` check, no count-vs-local-count comparison. If the read is from a slightly-lagged replica or hits a transient race where the server is mid-write, an appointment that exists locally (just edited, sync in flight) could be wiped. The tombstone filter (`isTombstoned`) protects against just-deleted rows, but not against just-edited ones.

**Fix:** before dispatching, diff the fresh snapshot against current `state.appointments`; only LOAD_STATE if the diff is non-trivial. Better: include `updated_at` in the diff so a server-side row older than the local copy doesn't clobber the local one.

### P2L2. FK type mismatch: `tickets.primary_manicurist_id` is `text`, `manicurists.id` is `uuid` ◻
`supabase/migrations/20260428200000_create_tickets.sql:30` references `manicurists(id)` which is defined as `uuid PRIMARY KEY` (`20260326012431_create_salon_schema.sql:48`).

Postgres normally rejects this — FK types must match. The migration's own header comment (lines 14-17) acknowledges the pattern: "external tables… store id as text, so the foreign keys here use text." That comment is correct for `queue_entries`, `appointments`, etc., but contradicts the actual `manicurists` schema. Either the FK constraint silently failed to attach (most likely — Postgres would reject the constraint at migration time but the migration uses `CREATE TABLE` which doesn't validate the FK target unless you have data, and an empty target table validates trivially), or the type-checking is loose because PostgREST does runtime casting.

Either way: there's no actual referential integrity between `tickets.primary_manicurist_id` and `manicurists.id`. Deleting a manicurist that has tickets would not get a "still referenced" error from the DB. The `ON DELETE SET NULL` clause in the migration is dead code.

**Fix:** either (a) normalize all manicurist references to `uuid` across the schema with a migration that casts and re-creates the FKs, or (b) accept the lack of integrity (it's been fine for a year) and remove the misleading FK clauses + `ON DELETE` modifiers so a future maintainer doesn't trust them.

### P2L3. Inconsistent `Sync failed` toast wording (informational only) ✅
Fixed earlier in this session — all 21 toasts now use a U+2014 em-dash. Noting here for the audit trail.

### P2L4. Phone/email columns are unbounded `text` with no CHECK ◻
`supabase/migrations/20260428200000_create_tickets.sql:28-29` — `client_phone text NOT NULL DEFAULT ''`, `client_email text NOT NULL DEFAULT ''`. Same on `appointments`. No max length, no regex, no E.164 enforcement. A typo (a one-character paste, an autocomplete glitch) is silently stored. Eventually this bites when SMS launches and a number like `''` or `'unknown'` reaches Twilio.

**Fix:** add CHECK constraints on insertion. For email, a minimal `(client_email = '' OR client_email ~ '^[^@]+@[^@]+$')` rejects the obviously-wrong shapes without enforcing strict RFC compliance.

## What was checked and came up clean

- **Reducer purity:** I read the entire 1442-line reducer. No `state.x.push(...)`, `state.x.splice(...)`, no in-place sorts of state slices. The `slice() + index assignment + spread` pattern at lines 1305, 1341, 1350, 1367, 1389, 1397, 1416, 1427 (and the `{...state}` + property write at line 1290) is the *canonical* Redux immutable-update idiom — the array/object being mutated is the fresh copy that gets returned, not the live state. Pass-2's first reducer agent flagged these as bugs; they're not.
- **Realtime channel cleanup:** the master effect at line ~1151 cleans up all 11 channels on unmount, gated correctly on `state.loaded`. No leaks.
- **Echo prevention:** `wasRemote` flag plus `apptTombstones` map cover the appointment loop. The pattern is consistent across the other tables (`isApplyingRemoteRef`).
- **send-push edge function:** auth & input validation are reasonable. The "VAPID public key prefix in debug output" the schema agent flagged is not a vulnerability — the public key is by design public. Dropped.
- **nightly-save-history retention purge:** the LT-vs-LTE concern was nitpicking — 1ms of boundary precision is irrelevant given the function only runs once a day.
